# 쿠폰 시스템 개편 가이드

> ugwanggiAPI 프로젝트의 쿠폰 시스템을 개편하기 위한 가이드 문서입니다. 현재 쿠폰 시스템의 구조와 한계를 분석하고, 퍼센티지 할인 방식을 도입하기 위한 구체적인 코드 수정 방향을 제시합니다.

- **프로젝트:** ugwanggiAPI (feature-new_coupon 브랜치)
- **핵심 변경:** 기존 고정 금액 쿠폰에 더해 퍼센티지(%) 할인 방식 추가
- **핵심 파일:** `subscribe/models.py`, `subscribe/views.py`
- **작성:** 2026-04-13 / 작성자: 김명승 (도움: Claude Opus 4.6)

## 1. 현재 쿠폰 시스템 개요

### 테이블 구조
하이픈의 쿠폰 관련 테이블은 세 개입니다. 이 중 실제로 동작하는 건 두 개뿐입니다.

**CustomCouponSettings (쿠폰 정책)** — 쿠폰의 "규칙서". 이 쿠폰을 쓰면 얼마를 내야 하는지, 몇 개월이 무료인지 같은 조건이 담깁니다.
```
테이블: ugwanggi_custom_coupon_settings
settings_id    (PK)       - 정책 고유 번호
status         (VARCHAR)  - 구독 유형 (basic, team)
name           (VARCHAR)  - 정책 이름
free_month_nb  (INT)      - 무료 제공 개월 수
amount         (INT)      - 할인 후 결제 금액
payment_left   (INT)      - 남은 결제 횟수
plan           (VARCHAR)  - 결제 주기 (month, year)
can_upgrade    (BOOL)     - 업그레이드 가능 여부
additional_user (INT)     - 추가 사용자 수
```

**CustomCoupon (쿠폰 코드)** — 사용자가 실제로 입력하는 쿠폰 코드. 각 코드는 Settings를 참조해 조건을 가져옵니다.
```
테이블: ugwanggi_custom_coupon
coupon_id       (PK)       - 쿠폰 고유 번호
coupon          (VARCHAR)  - 쿠폰 코드 문자열 (예: ABC12345)
settings_id     (FK)       - 어떤 정책을 따르는지
used            (BOOL)     - 사용 여부
expiration_date (DATE)     - 만료일 (있지만 검증 안 됨)
creator_user_id (INT)      - 충성도 쿠폰 생성자
user_id         (INT)      - 쿠폰을 사용한 사용자
```

**CustomCouponSettings2 (확장 정책 - 현재 미사용)** — Settings를 확장한 버전으로, 퍼센티지 할인율과 플랫폼별/플랜별 조건 필드가 추가되어 있습니다. **그러나 현재 어떤 뷰에서도 이 테이블을 참조하지 않습니다.** 모델만 정의되어 있고 결제 로직과 연결되지 않은 상태입니다. **이번 개편의 핵심은 이 잠든 테이블을 깨우는 것입니다.**
```
테이블: ugwanggi_custom_coupon_settings2
settings_id   (PK)       - 정책 고유 번호
name          (VARCHAR)  - 정책 이름
youtube       (BOOL)     - YouTube 플랫폼 적용 여부
instagram     (BOOL)     - Instagram 플랫폼 적용 여부
bundle        (BOOL)     - 번들 적용 여부
basic         (BOOL)     - Basic 플랜 적용 여부
team          (BOOL)     - Team 플랜 적용 여부
monthly       (BOOL)     - 월간 결제 적용 여부
yearly        (BOOL)     - 연간 결제 적용 여부
percentage    (INT)      - 할인율 (%)
payment_number (INT)     - 적용되는 결제 회차
can_upgrade   (BOOL)     - 업그레이드 가능 여부
```

### 현재 쿠폰 적용 흐름
```
사용자가 쿠폰 코드 입력
  -> CheckCustomCouponView: DB에서 코드 조회 (used=False인 것만)
  -> 쿠폰에 연결된 Settings 조회 (amount, plan, free_month_nb 확인)
  -> FirstPaymentView에서 결제 진행
       - free_month_nb > 0 이면 결제 생략
       - 아니면 Settings의 amount로 실제 결제
  -> 구독(Subscription) 레코드 생성 (amount: 쿠폰 가격 / plan: 쿠폰 주기)
  -> 쿠폰 사용 처리
       - 관리자 쿠폰: 해당 쿠폰만 used=True
       - 충성도 쿠폰: 같은 creator의 쿠폰 전부 used=True
```

### 현재 시스템의 한계
> **핵심 문제:** 현재 쿠폰은 "고정 금액 덮어쓰기"만 가능합니다. 원래 가격이 뭐든 쿠폰의 amount 값으로 대체하는 방식이라, 퍼센티지 할인 같은 유연한 처리가 불가능합니다.

- **만료일 검증 부재** — `expiration_date` 필드가 있지만 어디서도 확인하지 않아 만료된 쿠폰도 사용 가능
- **Settings2 미연결** — percentage, 플랫폼별/플랜별 조건 필드가 준비돼 있으나 결제 흐름에서 한 번도 참조하지 않음
- **쿠폰-구독 간 추적 불가** — 구독/결제 이력에 쿠폰 ID를 기록하지 않아 역추적 불가
- **경우의 수 폭발** — "30% 할인" 하나를 구현하려 플랜별/주기별/플랫폼별로 각각 Settings 필요. 조합이 곱셈으로 증가

## 2. 할인 방식 비교: 고정 금액 vs 퍼센티지

### 고정 금액 덮어쓰기 방식 (현재)
쿠폰 Settings에 금액이 직접 적혀 있고, 결제 시 원래 가격을 이 금액으로 "교체". "30% 할인" 쿠폰을 만들려면 플랜·주기·플랫폼 조합마다 Settings가 필요 → **10개 이상 필요**.
- 장점: 구현 단순(코드 변경 없이 현재 그대로 사용 가능), "1,000원 체험" 같은 특수 이벤트에 적합
- 단점: 할인 의도 하나당 Settings를 플랜 수만큼 생성, 가격 정책 변경 시 전부 재계산, **플랜 추가 시 경우의 수 곱셈 증가**

### 유연한 퍼센티지 방식 (개편안)
쿠폰에는 비율만 적고, 실제 할인 금액은 결제 시점에 계산. 같은 "30% 할인"이라면 **Settings2 1개(`percentage=30`)로 전 플랜 처리**.
- 장점: Settings 1개로 전 플랜 동일 할인, 가격 정책 변경에도 쿠폰 수정 불필요, 마케팅 안내 단순
- 단점: "정확히 1,000원" 같은 고정 가격 이벤트엔 부적합, Settings2 참조 로직이 없어 뷰 수정 필요

### 결론: 왜 퍼센티지 방식이 메인인가
> **퍼센티지는 1개로 N개의 일을 하고, 고정 금액은 N개로 N개의 일을 합니다.** 따라서 퍼센티지를 메인으로 채택하되, 고정 금액은 특수 상황의 보조 수단으로 유지합니다.

근거 세 가지: ① **개발 비용** (플랜 3 × 주기 2 × 플랫폼 3 = 고정 18개 vs 퍼센티지 1개) ② **가격 변동 대응** (퍼센티지는 비율만 있어 가격이 바뀌어도 무수정) ③ **마케팅 단순성** ("이 코드로 30% 할인" 한 문장).

고정 금액을 완전히 없애지 않는 이유: "1,000원 체험"(정확한 금액), "3개월 완전 무료"(amount=0이 직관적), 제휴사 고정 단가 계약.

## 3. 개편 방향: Settings2를 깨운다
**새 테이블을 만들 필요가 없습니다.** `CustomCouponSettings2`는 이미 percentage·플랫폼별·플랜별 플래그를 모두 갖췄습니다. 문제는 이 테이블을 읽는 코드가 없다는 것뿐. 개편의 핵심은 두 군데 수정:

```
수정 대상 1: CheckCustomCouponView (쿠폰 유효성 확인) - subscribe/views.py
  -> Settings2 기반 쿠폰도 처리하도록 분기 추가
수정 대상 2: FirstPaymentView (결제 처리) - subscribe/views.py
  -> 퍼센티지 할인 계산 로직 추가
```

## 4. 구체적 코드 수정 가이드

> ⚠️ 이 장은 "이대로 복붙하라"가 아니라, 개발자가 알아야 할 변수 흐름·수정 지점·주의사항을 정리한 것입니다. 실제 구현 시 최신 소스 기준으로 라인 번호와 변수명을 재확인하고, 반드시 테스트하세요.

### 4-1. 먼저 알아야 할 것: 결제 변수 흐름
`FirstPaymentView.post()`에서 **"돈을 얼마 받을지"를 결정하는 변수가 하나가 아니라 여러 개**라는 점이 핵심입니다.
```
[라인 856] calculatePrice() 호출
  반환값: amount(번들 전체), yt_amount(YouTube 단독), ig_amount(Instagram 단독),
          to_pay(실제 첫 결제 청구액), is_bundle(번들 여부)
[라인 872-878] ## COUPON GESTION
  현재 쿠폰이 바꾸는 변수:   amount, plan (덮어쓰기)
  현재 쿠폰이 안 바꾸는 변수: to_pay, yt_amount, ig_amount (원래 가격 그대로)
[라인 901]      free_month_nb 있으면 결제 스킵, 없으면 to_pay로 결제
[라인 902-909]  iamportPayment(..., to_pay, ...)  <- amount가 아닌 to_pay 사용
[라인 1047-1055] 구독 생성: 번들=amount / 비번들=yt_amount, ig_amount
```
> **핵심 주의:** 퍼센티지 할인을 적용하려면 `amount`만 수정해선 안 됩니다. `to_pay`, `yt_amount`, `ig_amount`까지 모두 할인율을 적용해야 실제 결제와 구독 금액이 일치합니다.

### 4-2. 모델 수정: 두 가지 선택지

**선택지 A: 기존 Settings 테이블에 필드 추가** (`subscribe/models.py`, `CustomCouponSettings`)
```python
discount_type = CharField(max_length=20, default="fixed")  # "fixed"(기존) | "percentage"(신규)
percentage    = IntegerField(default=0)                     # 예: 30이면 30% 할인
```
- 장점: FK 구조 안 건드림, 기존 쿠폰은 `discount_type="fixed"` 기본값으로 하위 호환 보장, 기존 코드 안 깨짐
- 단점: Settings2의 플랫폼별/플랜별 조건 필드를 활용 못 함

**선택지 B: Settings2를 FK로 연결** (`subscribe/models.py`, `CustomCoupon`)
```python
settings2 = ForeignKey(CustomCouponSettings2, on_delete=CASCADE,
                       db_column="settings2_id", null=True, blank=True)
# 주의: 필드명은 settings2 (settings2_id로 지으면 내부적으로 settings2_id_id가 됨)
```
- 장점: Settings2의 세분화 조건 필드 그대로 활용, 새 테이블 불필요
- 단점: `settings_id`가 NOT NULL이라 더미 레코드 필요 또는 nullable 변경 → 기존 `coupon.settings_id.xxx` 호출 전반에 None 체크 추가 필요

> **AI 권장:** 빠르고 안전하게 가려면 **선택지 A**. 플랫폼별/플랜별 세분화 제어가 꼭 필요하면 B를 택하되 `settings_id` nullable 변경 영향을 전수 점검.

### 4-3. 뷰 수정: CheckCustomCouponView (`subscribe/views.py`)
**① 만료일 검증 추가** (쿠폰 조회 직후)
```python
coupon = CustomCoupon.objects.filter(coupon=code, used=False).first()
if coupon is None:
    return 400
# --- 추가 ---
if coupon.expiration_date and coupon.expiration_date < date.today():
    return 400 ("만료된 쿠폰입니다")
```
**② 퍼센티지/고정 응답 분기** (선택지 A 기준)
```python
settings = coupon.settings_id
if settings.discount_type == "percentage":
    return Response({"type": "percentage", "percentage": settings.percentage,
                     "can_upgrade": settings.can_upgrade})
else:
    return Response({"type": "fixed", "amount": settings.amount, "plan": settings.plan, ...})
```
> 프론트엔드도 응답의 `type` 필드를 보고 "30% 할인" vs "70,000원" 표시를 분기해야 합니다.

### 4-4. 뷰 수정: FirstPaymentView (`subscribe/views.py`, 라인 872 부근) — 가장 중요
**수정 지점 1: 쿠폰 적용 블록 (라인 872-878)** — 퍼센티지면 "덮어쓰기"가 아니라 "비율 계산"
```python
settings = coupon.settings_id
if settings.discount_type == "percentage":
    rate = settings.percentage / 100
    amount    = round(amount * (1 - rate))     # 번들 금액
    yt_amount = round(yt_amount * (1 - rate))  # YouTube 금액
    ig_amount = round(ig_amount * (1 - rate))  # Instagram 금액
    to_pay    = round(to_pay * (1 - rate))     # 실제 청구 금액
    # plan은 사용자가 선택한 값 유지 (덮어쓰지 않음)
else:
    amount = settings.amount
    plan = settings.plan
```
> **왜 네 변수를 다 바꾸나:** amount=번들 구독(1047-1051), yt_amount=비번들 YouTube(1053), ig_amount=비번들 Instagram(1055), to_pay=실제 첫 결제(902-909). amount만 바꾸면 번들만 할인되고 비번들·실제 결제는 원래 가격으로 진행됩니다.

**수정 지점 2: 결제 여부 판단 (라인 901)** — 퍼센티지엔 free_month_nb 개념이 없으므로 분기
```python
should_charge = True
if coupon is not None:
    settings = coupon.settings_id
    if settings.discount_type == "fixed" and settings.free_month_nb:
        should_charge = False  # 무료 체험 쿠폰
    # 퍼센티지 쿠폰은 항상 결제 (할인된 to_pay 금액으로)
if should_charge and to_pay > 0:
    # 결제 수행
```
**수정 지점 3: 쿠폰 사용 처리 (라인 1066-1073)** — **수정 불필요.** `used` 플래그와 `creator_user_id`만 다뤄 타입 무관하게 동작.

### 4-5. 소수점 처리: round()를 사용할 것
```python
amount = int(99999 * 0.7)    # 69999 (1원 차이 발생 가능)  ← 잘못
amount = round(99999 * 0.7)  # 69999 (수학적으로 정확)      ← 올바름
```
결제 금액의 1원 차이는 정산 문제가 될 수 있으니 일관되게 `round()` 사용.

### 4-6. 마이그레이션 및 직렬화
- `python manage.py makemigrations subscribe` → `python manage.py migrate`
- `subscribe/serializers.py`에 새 필드(`discount_type`, `percentage`) 직렬화 포함
> `discount_type` default="fixed"로 두면 기존 Settings 레코드는 자동으로 고정 금액 타입이 되어 별도 데이터 마이그레이션 불필요.

### 4-7. 쿠폰 생성 API: CustomCouponView (`subscribe/views.py`)
- **선택지 A:** 별도 수정 거의 불필요. 단, `CustomCouponSettingsView.post()`에서 `discount_type`·`percentage`를 받을 수 있어야 함
- **선택지 B:** 요청에 `settings2_id` 추가, settings_id와 동시 입력 거부 검증, 생성 시 settings2 FK 저장
> 관리자: 정책 레코드 생성 → 쿠폰 코드 생성 순서 준수. 관리자 UI는 discount_type 선택에 따라 입력 폼 분기 필요.

- [ ] 새 쿠폰(Settings2 기반)의 퍼센티지 할인이 정확히 계산되는지 확인했는가
- [ ] 만료된 쿠폰이 정상적으로 거부되는지 확인했는가
