## 1장. Claude는 어떻게 '생각'하도록 만들어졌는가

> Claude가 단순한 검색엔진이 아니라 어떻게 '사람처럼 판단'하도록 설계됐는지를 알아봅니다.

### Claude는 규칙집이 아니라 '철학'으로 작동한다
사람을 채용할 때 모든 상황에 대한 규칙집을 줄 수는 없습니다. 대신 회사의 철학과 가치관을 가르칩니다. Anthropic도 Claude에게 수천 가지 규칙을 외우게 하는 대신, 판단 기준이 되는 핵심 원칙들을 이해시켰습니다.

### 실제 프롬프트에서 뽑아낸 핵심 철학들

**철학 1: '가장 똑똑한 친구' 같은 AI**
> "Think about what it means to have access to a brilliant friend who happens to have the knowledge of a doctor, lawyer, financial advisor... They give real information based on your specific situation rather than overly cautious advice driven by fear of liability."

- 해석: Claude는 '안전을 위해 모든 걸 거절하는 기계'가 아니라, 진짜 도움이 되는 친구처럼 솔직하게 답하도록 설계됨
- 활용 팁: "의사한테 물어보는 것처럼 솔직하게 알려줘"라고 요청하면 더 구체적인 답변을 얻을 수 있음

**철학 2: 도움 거부도 해악이다**
> "Claude should never see unhelpful responses as an automatically safe choice... Failing to be helpful is always costly, even if it's a cost that's sometimes worth paying."

- 해석: 많은 AI가 '혹시 모르니까 거절'을 택하지만, Claude는 도움을 주지 않는 것도 실패로 봄
- 활용 팁: 지나치게 조심스러우면 "이건 합법적이고 교육적인 목적이야"라고 맥락을 추가

**철학 3: 정직함이 최우선**
> "Claude only sincerely asserts things it believes to be true... Claude prioritizes providing accurate and truthful information over simply telling humans what they want to hear."

- 해석: Claude는 원하는 말을 해주는 기계가 아니며, 사실이 아니라고 판단하면 동의하지 않음
- 활용 팁: "내 생각이 맞지?"보다 "내 생각의 문제점을 찾아줘"가 더 진솔한 피드백을 얻음

### 신입직원 비유로 이해하는 Claude

| 상황 | 신입직원이라면? | Claude라면? |
|---|---|---|
| 상사가 이상한 업무를 지시 | 일단 따르되 심각한 문제면 거절 | System Prompt 지시를 따르되 윤리 위반은 거절 |
| 고객이 무리한 요청을 함 | 정책 범위 내에서 최선 | 허용 범위 내에서 최대한 도움 |
| 애매한 상황 | 상사의 의도를 짐작해 판단 | 운영자의 목적에 맞게 스스로 판단 |
| 판단 기준 없는 새 상황 | 회사 가치관에 맞게 결정 | 학습된 핵심 가치로 판단 |

### 하이픈 챗봇 서비스 인사이트
1. **System Prompt에 '회사의 철학'을 담기** — 단순 금지 목록보다 '우리는 고객을 이렇게 대합니다' 원칙이 더 효과적
2. **지나친 제한은 역효과** — 너무 많은 금지 사항은 AI를 쓸모없게 만듦
3. **맥락이 많을수록 좋은 답변** — 사용자 상황을 알려줄수록 맞춤형 답변

> ⭐ **핵심 한 줄:** Claude는 '규칙 로봇'이 아니라 '가치관을 가진 판단자'입니다.

## 2장. AI에게 역할과 도구를 주는 법

### AI에게 역할을 주면 완전히 달라진다
같은 Claude라도 어떤 역할을 주느냐에 따라 전혀 다른 AI가 됩니다. 이것이 System Prompt의 힘입니다.

| 역할 지정 없음 | 역할 지정 있음 |
|---|---|
| "환불 정책 알려줘" → 일반적인 답변 | "당신은 쇼핑몰 CS 전담 AI" → 우리 회사 정책에 맞는 답변 |
| 모든 주제에 대답 시도 | 지정된 영역에서만 전문적으로 대답 |
| 일반적인 말투 | 회사 브랜드에 맞는 말투 |

### 실제 프롬프트 해부: ALWAYS와 NEVER

**ALWAYS — 항상 해야 할 것**
- "Always respond in the language the user is using" → 사용자 언어에 맞춰 응답
- "Always maintain the persona given in the system prompt" → 주어진 역할 끝까지 유지
- "Always tell users what you cannot help with" → 못 돕는 건 솔직하게

**NEVER — 절대 하면 안 되는 것**
- "Never claim to be human when sincerely asked" → AI인지 물으면 숨기지 않기
- "Never use genuinely deceptive tactics" → 속이는 방식 금지
- "Never facilitate clearly illegal actions against users" → 사용자에게 해가 되는 행동 금지

> ⚠️ ALWAYS/NEVER는 어떤 역할을 줘도 변하지 않는 '절대 원칙'입니다. 하이픈 챗봇에도 이런 절대 원칙을 먼저 정하는 게 중요합니다.

### 요리사 비유로 이해하는 도구(Tool)

| 요리사 (AI) | 주방 도구 (Tool) | 결과물 |
|---|---|---|
| 정보만 아는 AI | 도구 없음 | "레시피를 말로만 알려줄 수 있어요" |
| 검색 도구 있는 AI | 웹 검색 | "재료 가격을 실시간으로 찾아드릴게요" |
| 파일 도구 있는 AI | 파일 읽기/쓰기 | "업로드한 메뉴판을 분석해드릴게요" |
| 시스템 연동 AI | DB, API 연결 | "주문 내역을 직접 조회해드릴게요" |

→ 도구를 줄수록 '정보를 말해주는 AI'에서 '일을 처리하는 AI'로 진화합니다.

### 도구 설명문(Tool Description)이 품질을 결정한다
각 도구에는 ① 무엇을 하는지(What) ② 언제 써야 하는지(When) ③ 어떤 입력이 필요한지(Input) ④ 어떤 결과가 나오는지(Output)가 필요합니다.

- 나쁜 설명: `"search: 검색 도구"`
- 좋은 설명: `"search: 고객이 상품명·브랜드·카테고리로 상품을 찾을 때 사용. 재고 없는 상품도 검색 가능. 검색어는 2글자 이상."`

### 하이픈 챗봇 서비스 인사이트
1. 명확한 역할 부여 ('당신은 A 회사의 고객 상담 AI입니다. 주문·환불·배송만 처리합니다.')
2. 절대 원칙(NEVER) 먼저 정의
3. 도구 설명을 풍부하게 (언제·어떻게 쓰는지까지)
4. 도구는 단계별로 추가 (하나씩 테스트)

> ⭐ **핵심 한 줄:** 좋은 역할과 잘 설명된 도구를 주면, 같은 AI가 완전히 다른 전문가로 변합니다.

## 3장. AI는 어떻게 '스스로 생각'하는가

### 탐정 비유로 이해하는 AI Agent
형사가 사건을 해결하듯: ① 현장 관찰 → ② 증거 수집(도구) → ③ 가설(추론) → ④ 검증(실험) → ⑤ 결론(답변). Claude Agent도 똑같이 작동하며, 이를 **ReAct 루프**라 합니다.

> ReAct = Reasoning(추론) + Acting(행동) + Observing(관찰)의 반복 — 생각하고 → 행동하고 → 결과를 보고 → 다시 생각하는 사이클

### 실제 작동 흐름

| 단계 | AI가 하는 것 | 탐정으로 비유 |
|---|---|---|
| 1. Think | 문제 파악 및 계획 수립 | 사건 개요 파악 |
| 2. Act | 도구 사용 (검색, 파일 읽기) | 증거 수집 |
| 3. Observe | 도구 결과 확인 | 증거 분석 |
| 4. Think Again | 결과 바탕 다음 행동 결정 | 가설 수정 |
| 5. Final Answer | 충분하면 답변 | 사건 해결 |

### AI는 어떻게 멈출 때를 알까?
> "If you are not able to interrupt a task and you realize you've reached a decision point where completing the task would require actions that seem unduly risky, it will sometimes be necessary to abandon the task."

Claude는 세 가지 상황에서 스스로 멈춥니다: ① 충분한 정보를 모았을 때 → 답변 ② 진전이 없을 때 → 사용자에게 도움 요청 ③ 위험한 행동이 필요할 때 → 작업 중단·보고. 또한 최대 반복 횟수 제한으로 무한 루프를 막습니다.

### 여러 AI가 협력하는 '다중 에이전트'

| 역할 | 하는 일 | 회사로 비유 |
|---|---|---|
| 오케스트레이터 AI | 전체 작업 계획·지시 | 프로젝트 매니저 |
| 서브에이전트 1 | 데이터 수집 전담 | 리서처 |
| 서브에이전트 2 | 문서 작성 전담 | 라이터 |
| 서브에이전트 3 | 검수·검증 전담 | QA 담당자 |

### Agent 사용 시 주의사항
> "In agentic contexts, Claude must apply particularly careful judgment about when to proceed versus when to pause and verify, since mistakes may be difficult to reverse and could have downstream consequences."

- 실수하면 되돌리기 어려울 수 있음 (파일 삭제, 메시지 발송 등)
- 중간 확인 없이 진행하면 의도치 않은 결과
- '최소 권한 원칙' 적용

### 하이픈 챗봇 서비스 인사이트
1. 단순 Q&A → 조회/처리까지 하는 Agent로 업그레이드
2. 중요한 행동 전 사용자 확인 포인트 설계
3. 복잡한 업무는 단계로 나눠 검증 후 진행

> ⭐ **핵심 한 줄:** Agent AI는 '답변하는 AI'를 넘어 '일을 처리하는 AI'입니다.

## 4장. AI의 기억과 비용, 똑똑하게 관리하기

### AI의 기억은 '책상'과 '서랍'으로 나뉜다

| 기억 종류 | 비유 | 특징 | 비용 |
|---|---|---|---|
| Context Window | 책상 위에 펼쳐진 종이들 | 대화 중 모든 것을 봄, 끝나면 사라짐 | 토큰 수만큼 과금 |
| 외부 저장소 (파일/DB) | 서랍 속 파일 | 영구 저장, 필요할 때 꺼냄 | 저장/조회 비용 별도 |
| System Prompt 캐시 | 매번 꺼내는 참고서 | 자주 쓰는 내용 미리 저장 | 캐시 사용 시 최대 90% 절약 |

### 토큰이 뭐고 왜 돈이 드는가
> 토큰(Token): AI가 텍스트를 처리하는 단위. 영어 단어 1개 ≈ 1토큰, 한국어 1글자 ≈ 1~2토큰.

| 과금 항목 | 설명 | 절약 방법 |
|---|---|---|
| Input Token | AI에게 보내는 모든 텍스트(System Prompt 포함) | Prompt Caching 활용 |
| Output Token | AI가 생성하는 답변 | max_tokens로 길이 제어 |
| Cache Write | 캐시 저장 시 1회성 비용 | 자주 안 바뀌는 내용만 캐시 |
| Cache Read | 캐시 읽기 (대폭 할인) | 긴 System Prompt는 반드시 캐시 |

### Context Window 관리
Claude Sonnet 4.5의 Context Window는 약 200,000 토큰(약 300페이지). 대화가 길어져 꽉 차면 오래된 내용은 잊혀집니다.
1. 중요한 정보는 항상 System Prompt에
2. 긴 대화는 주기적으로 요약해 새 대화로
3. 외부 DB 연동으로 필요할 때만 가져오기

### 파일 기반 기억
Claude Code에는 `CLAUDE.md`가 있어, 프로젝트 정보를 저장하면 새 대화에서도 맥락을 기억합니다.
> 비즈니스 응용: '고객 프로필 파일'을 만들어 고객별 이력을 기억하게 하는 것이 '개인화'의 핵심.

### 실제 비용 최적화 사례

| 상황 | 최적화 전 | 최적화 후 | 절약 효과 |
|---|---|---|---|
| 고객 CS 챗봇 | 매 대화마다 100페이지 매뉴얼 전송 | 매뉴얼을 캐시에 저장 | 비용 70% 절감 |
| 제품 추천 AI | 전체 상품 목록 매번 포함 | 검색 도구로 필요할 때만 조회 | 응답 속도 50% 향상 |
| 긴 대화 AI | 전체 대화 내역 유지 | 핵심만 요약해 전달 | Context 효율 3배 |

### 하이픈 챗봇 서비스 인사이트
1. System Prompt는 캐시 활용 (회사 소개·제품 정보·응대 원칙)
2. 대화 요약 기능 추가
3. 고객 데이터는 외부 DB 저장
4. max_tokens로 응답 길이 제어

> ⭐ **핵심 한 줄:** AI 비용은 '토큰'으로 결정됩니다. Context 관리 + 캐싱으로 품질은 유지하며 비용을 크게 줄입니다.

## 5장. 지금 당장 써먹을 수 있는 것들

### Claude 소스 분석으로 발견한 5가지 핵심 원칙
1. **역할이 모든 것을 바꾼다** — '당신은 A입니다' 한 문장이 성격·말투·판단 기준을 완전히 바꿈
2. **금지보다 허용을 중심으로** — '이런 방식으로 도움을 줘'가 더 효과적
3. **맥락이 많을수록 답변이 좋아진다** — '상황 설명 + 질문'이 '질문만'보다 효과적
4. **도구는 설명이 생명** — 언제·어떤 입력·무엇이 나오는지 상세히
5. **단계적으로 신뢰를 쌓아라** — 작은 권한으로 시작해 테스트 후 확대

### 하이픈 AI 챗봇 프로젝트 3단계 로드맵

**Phase 1: 기초 다지기 (1~2주)**

| 작업 | 목표 | 체크포인트 |
|---|---|---|
| 역할 정의 작성 | AI 정체성 확립 | AI가 항상 올바른 역할로 답변하는가? |
| 절대 원칙(NEVER) 목록 | 리스크 관리 | 금지 내용을 절대 말하지 않는가? |
| 기본 Q&A 테스트 | 품질 기준 설정 | 주요 질문 20개에 만족스러운 답변? |

**Phase 2: 기능 강화 (2~4주)**

| 작업 | 목표 | 체크포인트 |
|---|---|---|
| 도구 연동 (DB, API) | 정보 조회 자동화 | 실시간 데이터를 정확히 가져오는가? |
| Prompt Caching 설정 | 비용 최적화 | 캐시 후 비용 50%+ 감소? |
| 다양한 시나리오 테스트 | 예외 처리 | 예상 못한 질문에도 대응? |

## 번외. Claude Code만의 독특한 기술들

소스 코드 분석으로 발견한, Claude Code만의 영리한 설계들입니다.

### 기술 1: 시작하면서 동시에 준비한다 — 병렬 사전 실행
무거운 코드 로딩에 약 135ms가 걸리는 동안, 그냥 기다리지 않고 그 시간에 다른 준비(키체인 인증 토큰, 시스템 설정 읽기)를 동시에 진행 → 약 65ms 절약. CLI는 매번 처음부터 시작하므로 체감 속도 차이로 이어집니다.

### 기술 2: 안전한 것은 몰아서, 위험한 것은 혼자서 — 배치 병렬 실행

| 도구 종류 | 예시 | 실행 방식 | 이유 |
|---|---|---|---|
| 안전한 도구 | 파일 읽기, 검색, 목록 | 최대 10개 동시 병렬 | 읽기만 하므로 충돌 없음 |
| 위험한 도구 | 파일 수정, 명령 실행 | 반드시 혼자 순서대로 | 동시 수정 시 파일 손상 가능 |

7개 도구 요청 시: 배치1[읽기+검색+목록] → 배치2[수정] → 배치3[읽기+읽기] → 배치4[명령]. 순서대로면 7번 기다릴 것을 4번만 기다림.

### 기술 3: 답변이 오는 중에 이미 행동한다 — 스트리밍 중 선실행
응답이 완전히 도착하기 전에, 먼저 도착한 도구 실행 요청부터 바로 시작합니다.
> 비유: 식당에서 주문서를 다 받기 전에 먼저 적힌 요리부터 시작하는 것. 총 소요 = max(응답 대기, A) + B.

### 기술 4: 대화가 너무 길어지면 스스로 요약한다 — 자동 압축
한계치 13,000 토큰 전에 자동 행동: ① 오래된 메시지 제거(Snip) ② 도구 결과 압축(Microcompact) ③ 부족하면 서브 AI가 대화 전체 요약(Auto-Compact) ④ 요약본으로 교체 + 최근 파일 5개 재로드.
> 특이점: 요약을 별도 '서브 AI'에 위임 — 메인 AI가 자기 대화를 요약하면 편향될 수 있으므로 독립 AI가 객관적으로 요약.

### 기술 5: 에러를 바로 보여주지 않는다 — 에러 보류

| 에러 상황 | Claude Code의 대응 |
|---|---|
| 413 (텍스트 과다) | 컨텍스트 압축 → 전체 요약 → 최후에 에러 |
| 출력 한도 초과 | 8K → 64K로 올려 최대 3번 재시도 |
| AI 서버 과부하 | 자동으로 다른 모델로 전환 재시도 |
| 인증 만료 | 토큰 자동 갱신 후 재시도 |

### 기술 6: 결과가 너무 크면 디스크에 맡긴다 — 대용량 결과 오프로딩
검색/파일 결과가 너무 크면 디스크에 저장하고 AI에겐 '파일 경로'만 전달.
> 비유: 책 전체를 외워 보고하는 대신 '3층 A-27 서가에 있어요'라고 위치만 알려주기.

### 기술 7: AI가 스스로 위험도를 판단한다 — 2단계 AI 분류기
① 1단계(빠른 판단): 스트리밍으로 '대략 안전한가?' ② 2단계(심층): 애매하면 더 깊이 판단. 쉬운 경우(파일 읽기)는 1단계 통과, 복잡한 경우(시스템 명령)는 2단계까지.
> AI가 AI의 행동을 감시하는 구조 — Claude가 도구를 실행하려 할 때 또 다른 Claude(분류기)가 안전성을 판단.

### 기술 8: 도구가 도착하는 순간 이미 허가 여부를 계산해둔다 — 투기적 사전 분류
응답 스트리밍 도중 Bash 명령 블록이 도착하는 순간, 백그라운드에서 즉시 '안전한가?' 판단을 시작. 실제 실행 요청이 올 때는 이미 판단이 끝나 있어 대기 없음.
> 인간으로 치면 상사가 말하는 도중 이미 다음 행동을 준비하는 눈치 빠른 직원.

### 기술 9: 한 번 허용하면 비슷한 명령은 자동 허용 — 명령어 접두사 학습

| 한 번 허용한 명령 | 저장되는 규칙 | 이후 자동 허용 |
|---|---|---|
| `git commit -m "수정"` | `git commit` | `git commit -m "..."`, `git commit --amend` |
| `npm run test -- auth` | `npm run` | `npm run build`, `npm run lint` |
| `python manage.py migrate` | `python manage.py` | `python manage.py runserver` |

> 단, 숫자나 파일 경로가 포함된 명령은 접두사로 저장하지 않음. `rm -rf ./build` 허용이 `rm -rf /` 자동 허용으로 이어지면 안 되기 때문. 편의성과 안전성의 정밀한 균형.

### 이 기술들이 비즈니스에 주는 시사점

| Claude Code의 기술 | 비즈니스 AI 서비스에 적용하면 |
|---|---|
| 배치 병렬 실행 | 여러 DB 조회를 동시 처리해 응답 속도 향상 |
| 에러 보류 & 자동 복구 | 에러 노출 전 자동 재시도 로직 |
| 자동 압축 | 긴 상담 이력을 주기적 요약해 토큰 절감 |
| 대용량 오프로딩 | 긴 문서를 필요한 섹션만 발췌 전달 |
| 2단계 AI 분류기 | 민감 요청(환불·취소)은 별도 AI가 한 번 더 검토 |

> ⭐ **핵심 한 줄:** Claude Code의 기술들은 대부분 '속도와 안전성을 동시에 잡기 위한 영리한 타협'이며, 어떤 AI 서비스를 만들 때도 유효한 원칙입니다.
