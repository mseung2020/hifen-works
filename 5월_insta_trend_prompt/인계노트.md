# 트렌드 분석 워크플로우 v1 — Import 노트 & 인계 초안

## 1. 파일

- `트렌드_분석_워크플로우_v1.yml` — Dify import용 본 파일
- `맛집_푸드.csv` — 시연 인스타 입력 (필수)
- `두쫀쿠 조회수 상위 10개.csv` — 시연 유튜브 입력 (선택, TOP1=두쫀쿠 시연 시)
- `global_stopwords.txt` — 운영용 풀버전 불용어 (yml에는 압축본 인라인됨)

## 2. Import 절차

1. Dify → Studio → 우상단 **Import DSL file** → `트렌드_분석_워크플로우_v1.yml` 선택
2. Import 후 다음을 **반드시 확인**:

### 2-1. 플러그인 의존성 확인
워크플로우가 다음 플러그인을 사용합니다. 미설치면 설치:
- **OpenAI** (`langgenius/openai`) — gpt-5-mini용
- **Anthropic** (`langgenius/anthropic`) — claude-sonnet-4-5용
- **Tavily** (`langgenius/tavily`) — 웹서치용

### 2-2. 모델 노드 점검 (각 LLM 노드 클릭해서 확인)
| 노드 | 모델 이름 (yml에 지정된 값) | 확인할 것 |
|---|---|---|
| LLM1 - 트렌드 클러스터 추출 | `gpt-5-mini` | OpenAI 플러그인이 인식하는지 |
| 광고주 인사이트 - 1차 생성 | `gpt-5-mini` | 동일 |
| 광고주 인사이트 - 자기검증 | `claude-sonnet-4-5` | **만약 4.6 사용 가능하면 4-6으로 변경** |
| 트렌드 스토리 - 1차 생성 | `gpt-5-mini` | OpenAI 인식 확인 |
| 트렌드 스토리 - 자기검증 | `claude-sonnet-4-5` | 동일하게 4.6으로 업그레이드 권장 |

> **모델 이름이 빨간 줄로 떠도 당황하지 말 것** — Anthropic 플러그인 버전에 따라 정확한 모델 ID가 다릅니다. 노드 클릭 → 모델 드롭다운에서 사용 가능한 Sonnet 최신 버전 선택하면 됩니다.

### 2-3. Tavily 노드 점검
- 노드 클릭 → API 키 등록 (Tavily 대시보드에서 발급받은 키)
- 다른 파라미터(query, max_results, search_depth 등)는 yml에 이미 설정되어 있음
- 만약 노드가 깨져 import되면: 노드 삭제 → 새 Tool 노드 추가 → Tavily Search 선택 → query 변수에 `{{#클러스터_추출기.search_query#}}` 연결

### 2-4. 시작 노드의 불용어 디폴트값 확인
- yml에 인라인된 불용어는 **압축 버전** (운영용 풀버전은 `global_stopwords.txt`)
- 시연이 잘 작동하면 그대로, 더 정밀한 필터링 원하면 시작 노드 → `stopwords_text` 디폴트 → `global_stopwords.txt` 내용 전체로 교체

## 3. 시연 실행

### 입력
- **인스타그램 CSV**: `맛집_푸드.csv` 업로드
- **유튜브 매칭 CSV**: `두쫀쿠 조회수 상위 10개.csv` 업로드
- **카테고리 힌트**: `맛집/푸드`
- **TOP K**: `20`
- **샘플링 행 수**: `500`
- **글로벌 불용어 리스트**: 디폴트 그대로 (또는 풀버전 붙여넣기)

### 기대 결과
- TOP1 클러스터: **두바이 디저트** (또는 유사한 두쫀쿠 관련 메인키워드)
- 출력 변수 8개:
  - `advertiser_insight_final` — 광고주 인사이트 JSON (최종)
  - `trend_story_final` — 트렌드 스토리 JSON (최종)
  - `cluster_pack_json` — LLM에 들어간 데이터 패키지 (디버그용)
  - `all_clusters_summary` — TOP K 전체 트렌드 키워드 (LLM1 출력)
  - `verification_report` — 사실 검증 리포트
  - `matched_post_count`, `matched_video_count`, `pack_debug` — 디버그

## 4. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| LLM1이 JSON 아닌 마크다운으로 출력 | gpt-5-mini가 가끔 형식 어김 | 프롬프트 시스템 메시지 끝에 "JSON only, no markdown" 한 줄 더 추가 |
| 클러스터 추출기에서 `LLM1 JSON 파싱 실패` 디버그 메시지 | LLM1 출력에 ```json``` 코드블록 또는 잡문 섞임 | 코드는 이미 코드블록 벗기는 로직 포함. 그래도 실패하면 LLM1 raw 출력 확인 후 프롬프트 강화 |
| `인스타 필수 컬럼 누락` 에러 | CSV 컬럼명 차이 | `맛집_푸드.csv`의 헤더가 `post_id, user_id, dm_send_count, views, likes, comments, followers, folloing, posts, post_topic, description, target`인지 확인. (`folloing`은 오타지만 그대로 둠) |
| Tavily 결과가 비어있음 | API 키 미등록 또는 한도 초과 | Tavily 대시보드에서 키·잔량 확인 |
| 트렌드 스토리에 환각 핸들 등장 | 자기검증이 잡지 못함 | `verification_report`의 `failed` 배열 확인. 자기검증 LLM의 모델을 더 강한 것으로 (Sonnet 4.6 또는 Opus) |
| 자기검증이 멀쩡한 결과를 과하게 다시 씀 | Claude의 수정 욕구 | 자기검증 시스템 프롬프트의 "가능한 한 1차 생성의 톤·구조 보존" 부분 강조, temperature 0.1로 낮추기 |

# 백엔드 인계 문서 초안 (이 문서를 다듬어 전달)

## A. 개요
인스타그램 게시물 데이터 기반 트렌드 클러스터 분석 시스템. 카테고리별 TOP K 클러스터를 추출하고, 각 클러스터에 대해 두 종류의 LLM 산출물을 생성:
- **광고주 인사이트** — 마케팅 의사결정용 (왜뜨나/누구에/캠페인)
- **트렌드 스토리** — 시간 흐름 서사 (점화/확산/현재단계)

## B. 데이터 계약 (cluster_pack JSON)

LLM 두 개 모두 다음 스키마의 단일 JSON을 입력으로 받음. **이 스키마가 LLM과 백엔드 간의 계약입니다.**

```jsonc
{
  "category_hint": "맛집/푸드",
  "rank": 1,
  "cluster": {
    "main_keyword": "두바이 디저트",
    "sub_keywords": ["두바이 쫀득떡", "카다이프", "..."]
  },
  "instagram_stats": {
    "matched_post_count": 23,
    "total_dm": 580000,
    "total_views": 45000000,
    "total_likes": 1200000,
    "total_comments": 8400,
    "avg_efficiency_dm_per_view": 0.0129,  // nullable
    "median_followers": 124000
  },
  "impact_post": {  // nullable
    "post_id": "DVyJIl2yBfe",
    "user_handle": "@space_tamnik",
    "dm_send_count": 239000,
    "views": 22345908,
    "likes": 507000,
    "comments": 4136,
    "followers": 2057873,
    "description_excerpt": "..."   // 600자 이내
  },
  "top_posts_sample": [             // 최대 5개, dm 내림차순
    { "user_handle": "...", "dm": 0, "views": 0, "likes": 0,
      "description_excerpt": "..." }  // 400자 이내
  ],
  "youtube_enrichment": {
    "available": true,              // false면 아래 필드 무시
    "matched_video_count": 10,
    "earliest_video": { ... } | null,
    "timeline_summary": { "first_publish_date": "...", "last_publish_date": "...",
                          "videos_per_month": {"YYYY-MM": N} } | null,
    "top_videos_sample": [ ... ]    // 최대 5개, views 내림차순
  },
  "web_context": {
    "available": true,
    "search_query_used": "...",
    "results_summary": "...",       // 4000자 이내 raw 텍스트
    "source_urls": [ ... ]          // 최대 8개
  },
  "availability_flags": {
    "youtube_ignition_usable": true,
    "web_context_usable": true,
    "post_count_sufficient": true,
    "efficiency_calculable": true
  }
}
```

## C. LLM 스펙

### C-1. 광고주 인사이트
- **1차 생성**
  - 모델: GPT-5-mini (시연), 프로덕션 권장: GPT-5 풀모델 또는 Claude Sonnet 4.6
  - Temperature: 0.4
  - 시스템 프롬프트: `트렌드_분석_워크플로우_v1.yml` 노드 9 참조
  - 출력 스키마:
    ```json
    {
      "main_keyword": "...",
      "sections": {
        "why_trending":  {"title":"왜 뜨고 있나", "body":"..."},
        "who_resonates": {"title":"누구에게 통하나", "body":"..."},
        "campaign_use":  {"title":"캠페인에 어떻게 쓸까", "body":"..."}
      },
      "data_caveats": []
    }
    ```
- **자기검증**
  - 모델: Claude Sonnet 4.5 (시연), 프로덕션: Sonnet 4.6 또는 Opus
  - Temperature: 0.2
  - 7개 체크리스트 기반 (yml 노드 10 참조)

### C-2. 트렌드 스토리
- **1차 생성**
  - 모델·temperature: GPT-5-mini, 0.3
  - 출력 스키마:
    ```json
    {
      "main_keyword": "...",
      "sections": {
        "ignition":      {"title":"점화", "date_or_period":"...", "primary_actor":"...", "narrative":"..."},
        "spread":        {"title":"확산 경로", "narrative":"..."},
        "current_stage": {"title":"현재 단계", "phase_label":"...", "narrative":"...", "advertiser_recommendation":"..."}
      },
      "cited_facts": [ {"type":"...", "value":"..."} ],
      "data_caveats": []
    }
    ```
  - `phase_label` 7개 enum: `점화 단계, 초기 확산, 급상승 구간, 정점 진입, 정점 후 안정, 하강 진입, 양산화·포화`
- **사실 검증 (결정론적 Python)**
  - `cited_facts`의 각 항목이 `cluster_pack`에 실재하는지 매칭
  - handle: 부분문자열 매칭, date: 정확 매칭, stat: ±2% 톨러런스 숫자 매칭
  - 출력: `verification_report` (실패 항목 리스트), `fail_count`
- **자기검증 + 사실교정**
  - 모델: Claude Sonnet 4.5
  - `verification_report`의 실패 항목을 우선 처리 → 7개 체크리스트로 전반 점검

## D. 하네스 설계 의도 (인계 핵심 자산)

### D-1. 광고주 인사이트의 하네스 = 단일생성 + 자기검증
- **3섹션이 서로 강하게 연결됨** (누구에게 통하는지가 정해져야 캠페인 액션이 의미 있음)
- 분할정복(섹션별 별도 LLM 호출) 대신 **단일 호출 + 자기검증 1패스**로 충분
- 자기검증의 핵심 가드: 데이터 환각, 카테고리 하드코딩, 광고 클리셰, 결측 정직성

### D-2. 트렌드 스토리의 하네스 = 단일생성 + Python검증 + 자기검증
- **사실 인용이 본업**이라 환각 리스크가 가장 큼
- LLM의 자기검증만으론 "이 숫자가 진짜 있나"를 빠뜨릴 수 있어 **결정론적 후처리 검증**을 추가
- `cited_facts` 배열을 LLM이 enumerate → Python이 cluster_pack과 매칭 → 실패 시 자기검증 LLM이 수정
- 이게 이 시스템의 신뢰성 핵심 메커니즘

### D-3. 결측 데이터 처리 — `availability_flags` 패턴
- 모든 옵셔널 데이터에 명시적 boolean 플래그를 둠
- 프롬프트가 "flag가 false면 ~ 처리" 형태로 분기
- LLM이 "데이터 없는 영역을 추측으로 채우는" 환각 패턴을 구조적으로 차단
- 데이터 한계는 `data_caveats` 배열에 정직히 기록

### D-4. 카테고리 무관성 (cluster-agnostic)
- 프롬프트 어디에도 "맛집" "패션" 같은 카테고리 하드코딩 없음
- `category_hint`를 변수로 받아 LLM이 자기 문맥화
- 자기검증 체크리스트 2번에서 "카테고리 하드코딩 검출"을 명시적으로 점검

## E. 알려진 한계 / 향후 작업

1. **임팩트 게시물 = dm 최댓값 단일 선정** — 광고/협찬 게시물이 1위로 잡힐 리스크 있음. 향후: followers 가중치 보정 또는 LLM 기반 광고성 필터링
2. **클러스터 매칭 = 단순 부분문자열** — 의미적으로 유사하지만 표면형이 다른 게시물 누락 가능. 향후: 임베딩 유사도
3. **post_count_sufficient 임계 = 5** — 카테고리·시기에 따라 동적 조정 필요할 수 있음
4. **트렌드 스토리의 "확산 경로"는 행위자 수 기반** — 진짜 인플루언서 그래프(누가 누구를 보고 따라했는지)는 미반영. 향후: SNS 인용 그래프 데이터 통합
5. **시연에서는 TOP 1 클러스터만 깊은분석** — 실서비스에선 TOP K 전체에 적용 (잡 큐로 병렬 처리)
