6. AI가 만든 흔적
자각 못했을 수 있는데, 이 문서는 GPT로 만든 컨설팅 슬라이드 냄새가 명확히 난다. 구체적으로:

"다음 조치" 배지 + .why 고정 쌍: 데이터 품질 6개 섹션이 전부 같은 골격(제목 → why → next-step → items). 사람이 쓰는 보고서는 이렇게 정갈하게 안 맞아떨어진다. "유령 모델 2건"에 "다음 조치"가 있고 "자기참조 n건"에도 같은 배지가 있는데, 자기참조는 "별도 조치 필요 없음"이라고 쓴다(app.js:1161) — 배지 의미가 깨짐.
"이 앱이 기대는 곳 / 이 앱에 기대는 곳" 대구(app.js:380-391). 의도적이지만 과함. "outbound 의존 / inbound 의존"이 더 정확하고 개발자에게도 친절.
"한 장의 도면", "누가 기대고 있으며, 무엇이 방치되어 있는지"(app.js:106) — LLM이 좋아하는 트리콜론 + 대구. 한국어 문어의 기본 수준을 넘어 시적이다.
"같은 식구가 서로를 모르는 상태"(app.js:1139) — 의인화. 원칙의 "전문용어 회피"는 맞지만 의인화는 정확성 손실. "도메인 이름은 같지만 FK로 연결되어 있지 않은 테이블들"이면 충분.
border-left: 3px solid var(--blue) + 파란 배경 info 박스(styles.css:286, styles.css:723) — 2023년 이후 LLM이 생성하는 React 대시보드의 90%가 이 스타일.
"비싼 좀비"(app.js:905) — AI 네이밍. 한국어 팀 용어로는 "미사용 복합 API" 같은 중립이 더 전문적. "좀비"는 뚜렷한 색을 입히려다 과한 경우.
가장 AI 티가 나는 한 곳만 꼽으라면: 홈의 "이 시스템에서 지금 눈여겨볼 것"이라는 섹션 제목 + 3개 카드. 제목이 너무 "AI 도우미"스럽다. "핵심 소견" 또는 "눈에 띄는 신호" 정도로 중성화.

7. 이 문서의 진짜 한계 (제작자가 자각 못했을 만한)
"재탐색 없이 답한다"는 목표와 데이터 밀도가 어긋남. AI가 이걸 학습해도, 테이블 상세는 모달 안에 숨어있고 JS 렌더링이기 때문에 HTML 그대로 인덱싱하면 내용이 안 잡힌다. 지금 구조로는 "테이블 XXX의 컬럼이 뭐야?"에 AI가 답하려면 site_data.json을 별도로 읽어야 한다. 즉, HTML은 사람이 보는 용이고 AI 학습용으로는 JSON을 함께 주거나, 모달 내용을 SSG로 정적 섹션으로 펼쳐 두어야 한다.

검색 불가능. 홈에서 "X 테이블이 어디서 쓰여?" 한 번에 답하는 전역 검색이 없다. 테이블 탐색기로 가서 필터를 걸어야 함. PM 입장에서는 "그거 열어보고 싶은데 어딨는지 모르겠다"가 된다.

CDN 의존의 취약성. Pretendard, JetBrains Mono, d3 — 방화벽 환경·비행기·카페 WiFi에서 그래프 뷰 2개가 완전히 죽는다. "더블클릭 = 즉시 동작"이 실제로는 네트워크 있어야 동작. d3는 번들할 만한 크기(~280KB)다.

"폐쇄 세계" 가정이 런타임 상태와 분리되어 있다. 스케줄러(Celery, cron), 관리자 스크립트, Django admin 액션, shell을 통한 수동 호출, 마이그레이션 등은 전부 "프론트엔드 호출 스캔" 밖이다. About에 "크론 참조"라는 엣지 타입이 있긴 하지만(app.js:286), Celery task들과 admin action은 집계 범위 밖으로 보인다. "UNCALLED"라고 못 박힌 엔드포인트가 실은 Celery에서 호출되는 경우는?

"Confidence 배지"가 홈에서는 사라진다. 원칙은 "모든 수치에 confidence"인데, 홈 BigNum 4개에 배지 없음. 테이블 모달·엔드포인트 테이블에만 있음. "517개 테이블"이 HIGH인지 MED인지 문서가 자기 원칙을 지키지 않음.

텍스트/데이터 분리가 불완전. "517"이 본문 문자열로(app.js:422, app.js:207), "636"도 문자열로(app.js:209, app.js:867) 여러 곳에 박혀 있다. 템플릿 리터럴 아닌 고정 문자열. 데이터 갱신 시 수동으로 바꿔야 함 — 정적 분석 산출물의 본질과 모순.

Site_data가 1.5MB. 단일 HTML이 1.6MB인 가장 큰 이유. 초기 파싱·JSON.parse 비용. 모바일 첫 로딩 체감 가능. 홈 뷰는 META 수치 10여 개만 필요한데 매번 1.5MB를 메모리에 올림.

8. 만약 하나만 고친다면
홈 최상단에 5줄짜리 "이 문서 설명서" 블록을 추가. (섹션 3-1의 제안과 동일)

이유: 이 HTML의 최종 소비자가 "내부 전용 AI"라고 명시했는데, 현재 문서는 자신이 무엇인지 1인칭으로 선언하는 문장이 없다. AI든 사람이든 첫 5초 안에 "이건 2026-04-22 스냅샷의 정적 분석 결과이고, 폐쇄 세계 가정 하에서만 유효하고, 독자는 A/B/C"를 알아야 나머지 모든 숫자를 옳게 해석한다. 현재는 네비 탭 옆에 "ugwanggi architecture atlas"라는 영문 부제만 있을 뿐, 독자 정의·스냅샷 정의·가정의 강도·"이 문서를 쓰지 않아야 하는 경우" 가 홈에서 빠져 있다.

PM이 10초 안에 "아, 이건 정적 분석 리포트구나"를 잡는다.

나머지 모든 수치가 "폐쇄 세계 가정 하의 값"이라는 프레임을 자연스럽게 상속받는다.
